# Backrooms Mod для Minecraft — текст для презентации

---

## 1. Введение и концепция проекта

Данный проект представляет собой модификацию (мод) для игры Minecraft версии 1.20.1, разработанную на платформе Fabric. Мод реализует концепцию Backrooms — интернет-феномена, описывающего бесконечный лабиринт пустых офисных помещений с жёлтыми стенами, влажным ковром на полу, гудящими люминесцентными лампами и жуткими обитателями.

Игровой процесс устроен следующим образом: игрок создаёт обычный мир Minecraft и играет как обычно. Через двадцать секунд после захода в мир экран начинает плавно темнеть. Когда экран становится полностью чёрным, игрок автоматически телепортируется в отдельное измерение — The Backrooms. После телепортации экран плавно осветляется, и игрок оказывается в бесконечном лабиринте, в котором бродят два типа враждебных существ: быстрый Lurker и медленный, но мощный Howler.

Проект написан полностью на языке программирования Java версии 17 и содержит 15 Java-классов, 12 JSON-конфигурационных файлов, текстуры блоков и сущностей в формате PNG, а также 10 юнит-тестов.

## 2. Технологический стек и зависимости

### 2.1, Minecraft и Fabric

Minecraft Java Edition — игра, написанная на Java. Её исходный код обфусцирован (все имена классов и методов заменены на бессмысленные буквы типа `a`, `b`, `c`). Чтобы писать моды, нужна система, которая «расшифровывает» эти имена обратно в человекочитаемые. Для этого используется проект Yarn Mappings — набор маппингов, который переводит, например, обфусцированное имя `abc` обратно в `PlayerEntity` или `getHealth()`.

Fabric — это легковесный загрузчик модов для Minecraft. В отличие от другого популярного загрузчика Forge, Fabric работает на основе технологии Mixin, которая позволяет модифицировать код Minecraft на лету, без замены исходных файлов. Fabric Loader версии 0.15.7 загружает наш мод при старте Minecraft.

Fabric API версии 0.92.0 — это набор из более чем пятидесяти мини-модулей, каждый из которых предоставляет определённый API. Из них мы используем: fabric-lifecycle-events-v1 для тиковых событий сервера, fabric-networking-api-v1 для отправки сетевых пакетов между сервером и клиентом, fabric-object-builder-v1 для создания блоков и сущностей, fabric-client-rendering-v1 для отрисовки HUD-оверлея и регистрации рендереров сущностей.

### 2.2. Система сборки Gradle

Gradle версии 8.8 — это система автоматизации сборки проектов на JVM. В нашем проекте Gradle выполняет несколько критических функций.

Во-первых, управление зависимостями. Gradle автоматически скачивает Minecraft, Yarn Mappings, Fabric Loader, Fabric API и JUnit 5 из Maven-репозиториев. Все версии зависимостей вынесены в файл `gradle.properties`, что позволяет менять их в одном месте.

Во-вторых, компиляция и ремаппинг. Плагин Fabric Loom версии 1.7.4 (подключается через `build.gradle`) автоматически декомпилирует Minecraft, применяет маппинги Yarn и позволяет нашему коду ссылаться на классы Minecraft по понятным именам. При сборке Loom выполняет обратную операцию — ремаппинг нашего кода в обфусцированные имена, чтобы он работал с реальным Minecraft.

В-третьих, запуск и тестирование. Команда `gradlew runClient` запускает тестовый экземпляр Minecraft с нашим модом. Команда `gradlew build` компилирует мод и создаёт готовый JAR-файл в папке `build/libs/`. Команда `gradlew test` запускает юнит-тесты через JUnit 5.

Файл `build.gradle` содержит конфигурацию плагинов, зависимостей. Секция `dependencies` указывает на четыре основные зависимости плюс JUnit для тестов. Секция `test` включает `useJUnitPlatform()`, что активирует движок JUnit 5 Jupiter.

### 2.3. Файл `fabric.mod.json`

Это манифест мода — JSON-файл, который Fabric Loader читает при загрузке. В нём указан уникальный идентификатор мода (`"id": "backrooms"`), версия, совместимость с Minecraft и, самое важное, точки входа. Точка входа `"main"` указывает на класс `BackroomsMod`, который запускается и на сервере, и на клиенте. Точка входа `"client"` указывает на класс `BackroomsModClient`, который запускается только на клиенте. Разделение необходимо потому, что на выделенном (dedicated) сервере нет графики, звуков и HUD, и попытка загрузить клиентские классы приведёт к аварийному завершению.

## 3. Архитектура проекта и точки входа

### 3.1. Серверная точка входа — `BackroomsMod.java`

Класс `BackroomsMod` реализует интерфейс `ModInitializer`, метод `onInitialize()` которого вызывается Fabric Loader при старте. В этом методе в строго определённом порядке регистрируются все компоненты мода:

Первым шагом вызывается `ModBlocks.register()`, который регистрирует три кастомных блока в глобальном реестре Minecraft. Вторым шагом вызывается `ModItems.register()`, который регистрирует соответствующие предметы для этих блоков. Третьим — `ModSounds.register()`, регистрирующий три звуковых события. Четвёртым — `ModEntities.register()`, регистрирующий два типа сущностей и их атрибуты. Пятым — `ModDimensions.register()`, регистрирующий кодек кастомного генератора чанков. Шестым — `ModNetworking.registerC2SPackets()`, зарезервированный для будущих пакетов от клиента к серверу. И седьмым — `BackroomsTeleportHandler.register()`, подключающий тиковый обработчик телепортации.

Порядок регистрации важен: блоки должны существовать до предметов (так как `BlockItem` ссылается на блок), а сущности — до измерений (генератор ссылается на типы сущностей).

### 3.2. Клиентская точка входа — `BackroomsModClient.java`

Класс `BackroomsModClient` реализует интерфейс `ClientModInitializer`. В методе `onInitializeClient()` регистрируются: приёмник сетевых пакетов от сервера (fade-эффект), HUD-оверлей для отрисовки затемнения, фоновые зацикленные звуки, модельный слой для кастомной модели Lurker из Blockbench, и рендереры обоих сущностей. Рендерер — это класс, который определяет, как сущность выглядит в 3D: какую модель использовать и какую текстуру на неё натянуть.

## 4. Создание блоков

В Minecraft каждый блок — это экземпляр класса `Block` (или его наследника). Наш мод создаёт три кастомных блока в классе `ModBlocks.java`.

Первый блок — `BACKROOMS_WALL` (стена лабиринта). Создаётся вызовом `FabricBlockSettings.create()` с параметрами `strength(50.0f, 1200.0f)`. Первый аргумент — hardness (насколько долго ломается), второй — resistance (устойчивость к взрывам). Для сравнения: камень имеет hardness 1.5, а обсидиан — 50. Наша стена по прочности равна обсидиану, но всё же разрушима. Также указан `.requiresTool()`, что означает — без кирки блок невозможно добыть.

Второй блок — `BACKROOMS_FLOOR` (пол). Его hardness равен `-1.0f`, а resistance — `3600000.0f`. Значение `-1.0` — это специальная магическая константа в Minecraft, означающая «неразрушимый блок». Именно так устроен бедрок. Никакой инструмент, даже с зачарованием Efficiency V, не может сломать блок с hardness -1. Также применён `.dropsNothing()` — при разрушении (например, командой) ничего не выпадает.

Третий блок — `BACKROOMS_CEILING` (потолок). Так же неразрушим, как пол, но дополнительно имеет параметр `.luminance(state -> 12)`. Это лямбда-функция, которая принимает состояние блока (`BlockState`) и возвращает уровень света от 0 до 15. Мы всегда возвращаем 12 — это яркий свет, немного слабее факела (14) и глоустоуна (15). Звуковая группа указана как `BlockSoundGroup.GLASS`, чтобы при ходьбе по потолку (в креативном режиме) раздавался стеклянный звук, имитирующий люминесцентные панели.

Каждый блок регистрируется в глобальном реестре через `Registry.register(Registries.BLOCK, identifier, block)`. Идентификатор состоит из двух частей, разделённых двоеточием: пространство имён (namespace) и путь (path). Например: `backrooms:backrooms_wall`.

Помимо регистрации как блока, каждый блок должен быть зарегистрирован также как предмет через класс `BlockItem` в `ModItems.java`. Без этого блок существовал бы в мире, но его невозможно было бы держать в руках, ставить или находить в креативном инвентаре.

## 5. Создание сущностей

Сущности (entities) — это все движущиеся объекты в Minecraft: игроки, мобы, стрелы, лодки. Наш мод создаёт два враждебных моба: Lurker и Howler.

### 5.1. Иерархия наследования

Оба наших моба наследуют от класса `HostileEntity`, который является частью цепочки наследования Minecraft: `Entity → LivingEntity → MobEntity → PathAwareEntity → HostileEntity`. На каждом уровне добавляются новые возможности: `LivingEntity` даёт систему здоровья и урона, `MobEntity` даёт систему AI-целей, `PathAwareEntity` даёт навигацию по миру (pathfinding), `HostileEntity` делает моба враждебным.

### 5.2. Система искусственного интеллекта (Goals)

Minecraft использует систему целей (Goals) для управления поведением мобов. Каждая цель имеет числовой приоритет: чем меньше число, тем выше приоритет. Каждый тик (двадцатая часть секунды) Minecraft проверяет все цели и выполняет ту, чей приоритет наивысший и чьё условие выполнено.

Для Lurker определены следующие цели с приоритетами. Приоритет 0 — `SwimGoal`, не позволяет утонуть. Приоритет 1 — `MeleeAttackGoal` с множителем скорости 1.2, атакует в ближнем бою. Приоритет 2 — `WanderAroundFarGoal` со скоростью 0.8, бродит когда нет цели. Приоритет 3 — `LookAtEntityGoal`, смотрит на игрока в радиусе 16 блоков. Приоритет 4 — `LookAroundGoal`, случайно оглядывается.

Также определены две цели таргетинга: `RevengeGoal` (атакует того, кто ударил) и `ActiveTargetGoal` (активно ищет игрока в радиусе обнаружения).

### 5.3. Атрибуты сущностей

Атрибуты задаются через метод `createLurkerAttributes()`, возвращающий `DefaultAttributeContainer.Builder`. Lurker имеет 20 единиц здоровья (10 сердец), атаку 6, скорость 0.38 (обычный зомби — 0.23, то есть Lurker на 65 процентов быстрее зомби и значительно быстрее бегущего игрока), радиус обнаружения 40 блоков и 2 единицы брони.

Howler имеет 40 единиц здоровья (20 сердец), атаку 10, скорость 0.2 (медленнее зомби), радиус обнаружения 48 блоков, 6 единиц брони и 0.5 устойчивости к отбрасыванию.

### 5.4. Регистрация и ограничения спавна

Регистрация каждого типа сущности происходит через `FabricEntityTypeBuilder.create()`, где указывается категория спавна (`SpawnGroup.MONSTER`), ссылка на конструктор, размеры хитбокса (`EntityDimensions.fixed(ширина, высота)`) и дальность отслеживания в чанках.

Также регистрируется `SpawnRestriction` — ограничение спавна. По умолчанию враждебные мобы спавнятся только в темноте (уровень света менее 7). Но в Backrooms всегда светло (потолок излучает свет 12). Поэтому мы передаём лямбду `(type, world, reason, pos, random) -> true`, которая разрешает спавн при любом уровне света.

### 5.5. Кастомная модель Lurker

Модель Lurker создана в программе Blockbench — бесплатном онлайн-редакторе 3D-моделей для Minecraft. Blockbench экспортирует модель как Java-код, который затем адаптируется для работы с Fabric.

Класс `LurkerModel` наследует от `EntityModel<LurkerEntity>` и содержит шесть анимируемых частей: тело, голова, левая и правая ноги, левая и правая руки. Каждая часть — это объект `ModelPart`, состоящий из кубоидов (прямоугольных параллелепипедов) с UV-координатами для наложения текстуры.

Метод `setAngles()` вызывается каждый кадр рендеринга. В нём вычисляется анимация ходьбы: ноги и руки качаются вперёд-назад по оси pitch (X-ось) с использованием функции `MathHelper.cos()`, которая создаёт плавное синусоидальное движение. Руки двигаются в противофазе с ногами, как при естественной ходьбе человека. Голова поворачивается в сторону цели через параметры `netHeadYaw` и `headPitch`.

Модельный слой регистрируется через `EntityModelLayerRegistry.registerModelLayer()`, а рендерер Lurker создаётся как наследник `MobEntityRenderer`, принимающий экземпляр `LurkerModel` в конструкторе.

## 6. Создание измерения и генерация мира

### 6.1. Структура измерения в Minecraft

Каждое измерение в Minecraft состоит из трёх JSON-файлов. Первый — `dimension_type` (тип измерения), определяющий физические свойства: есть ли солнечный свет, работают ли часы и компас, какова фоновая освещённость, минимальная и максимальная высота мира. Для Backrooms мы отключаем солнечный свет (`has_skylight: false`), включаем потолок (`has_ceiling: true`), устанавливаем фоновый свет 0.5 и разрешаем спавн монстров при любом освещении.

Второй файл — `dimension` (само измерение), который связывает тип измерения с генератором мира. Указывается наш кастомный генератор `backrooms:backrooms_chunkgen` и источник биомов, фиксированный на единственный биом `backrooms:backrooms_biome`.

Третий файл — `biome` (биом), определяющий цвета неба, тумана, воды, а также какие мобы спавнятся естественным образом. Для Backrooms все списки спавна пусты, потому что мы управляем спавном вручную через код генератора.

### 6.2. Регистрация генератора через CODEC

Minecraft использует систему кодеков (Codec) из библиотеки Mojang DataFixerUpper для сериализации и десериализации объектов в JSON и обратно. Наш `BackroomsChunkGenerator` объявляет публичный `CODEC`, который описывает, как создать экземпляр генератора из JSON-данных. При загрузке мира Minecraft читает файл `dimension/the_backrooms.json`, встречает `"type": "backrooms:backrooms_chunkgen"`, находит зарегистрированный по этому ID кодек и создаёт наш генератор.

### 6.3. Как работает класс `BackroomsChunkGenerator`

Класс наследует от абстрактного `ChunkGenerator` и переопределяет семь методов. Главный из них — `populateNoise()`, который вызывается Minecraft для каждого нового чанка (участка мира 16×16 блоков).

В методе `populateNoise()` выполняется итерация по всем 256 позициям чанка (16 × 16). Для каждой позиции устанавливается пол на уровне Y=0 (неуязвимый блок `BACKROOMS_FLOOR`), потолок на уровне Y=6 (неуязвимый блок `BACKROOMS_CEILING` со светом), а между ними (Y=1 до Y=5) — либо стена, либо воздух, в зависимости от результата функции `isWall(worldX, worldZ)`.

### 6.4. Алгоритм генерации лабиринта — четырёхслойная сетка

Функция `isWall()` определяет, является ли данная мировая координата стеной. Она использует четыре слоя сеток, наложенных друг на друга.

Первый слой — основные стены с шагом сетки 32 блока и толщиной стен 2 блока. Они создают большие залы размером примерно 30×30 блоков. В каждой стене вырезано от одного до трёх дверных проёмов шириной 5 блоков. Эти стены присутствуют всегда и формируют основной каркас лабиринта.

Второй слой — средние стены с шагом 16 блоков и толщиной 1 блок. Они делят большие залы на более мелкие помещения и длинные коридоры. Но они существуют только в 60 процентах секторов — остальные секторы остаются открытыми залами. Решение о наличии стены принимается на основе хеш-функции от координат сектора.

Третий слой — мелкие перегородки с шагом 8 блоков. Они создают офисные кабинки и маленькие комнатки. Но эти перегородки появляются только в 20 процентах зон, создавая редкие «плотные» участки лабиринта.

Четвёртый слой — одиночные декоративные колонны, расставленные по сетке с шагом 10 блоков и смещением 5. Только 25 процентов возможных позиций заняты колоннами.

### 6.5. Хеш-функция SplitMix64

Для определения позиций проходов и наличия стен используется детерминистическая хеш-функция `mixHash(seed, a, b)`, основанная на алгоритме SplitMix64. Она принимает зерно (seed) и две координаты, и возвращает 64-битное число, которое равномерно распределено. Главное свойство — одни и те же координаты всегда дают одинаковый результат, независимо от порядка генерации чанков. Это гарантирует, что лабиринт не меняется при повторном посещении ранее генерированных чанков.

### 6.6. Проходы в стенах

Функция `isDoorway()` определяет, является ли конкретный блок стены дверным проёмом. Для каждой стены вычисляется от одного до трёх проходов. Позиция каждого прохода определяется хешем от координат ячейки сетки, что делает расположение дверей предсказуемым (детерминированным), но кажущимся случайным. Первый проход гарантирован, второй появляется с вероятностью 60 процентов, третий — 30 процентов и только для больших стен (32 блока).

## 7. Механика телепортации — класс `BackroomsTeleportHandler`

Телепортация реализована как конечный автомат (state machine) с пятью состояниями, управляемый серверным тиковым событием `ServerTickEvents.END_SERVER_TICK`.

Состояние 0 — отсчёт. Сервер считает тики для каждого игрока. Каждый игрок отслеживается через его UUID в двух HashMap: `playerState` (текущее состояние) и `playerTicks` (количество прошедших тиков). Когда счётчик достигает 400 тиков (20 секунд), сервер отправляет клиенту пакет fade-in и переходит в состояние 1.

Состояние 1 — ожидание затемнения. Сервер ждёт 40 тиков (2 секунды), пока на клиенте экран полностью затемнится. После этого вызывается метод `teleportToBackrooms()`, который получает объект мира Backrooms через `server.getWorld(ModDimensions.BACKROOMS_LEVEL_KEY)` и вызывает `player.teleport(backroomsWorld, 3.5, 1.0, 3.5, yaw, pitch)`.

Состояние 2 — пауза после телепортации. Сервер ждёт 20 тиков (1 секунду), давая время на загрузку чанков нового измерения. Затем отправляет пакет fade-out.

Состояние 3 — ожидание осветления. Ещё 40 тиков на анимацию.

Состояние 4 — завершено. Игрок в Backrooms, никаких дальнейших действий.

Если игрок уже в Backrooms при подключении к серверу (например, перезапустил клиент), автомат сразу переходит в состояние 4, чтобы не телепортировать повторно.

При отключении игрока из HashMap удаляется счётчик тиков, но не состояние — чтобы при переподключении не начинать отсчёт заново.

## 8. Сетевые пакеты — клиент-серверное взаимодействие

Minecraft работает по клиент-серверной архитектуре даже в одиночной игре. Встроенный сервер управляет логикой, а клиент отвечает за графику. Поэтому для эффекта затемнения экрана нужен сетевой пакет от сервера к клиенту.

Класс `ModNetworking` содержит идентификатор пакета `FADE_PACKET_ID` (`backrooms:fade_effect`) и два метода.

Метод `sendFadePacket()` выполняется на сервере. Он создаёт буфер (`PacketByteBuf`), записывает в него один boolean (true — затемнение, false — осветление) и отправляет конкретному игроку через `ServerPlayNetworking.send()`.

Метод `registerS2CPackets()` выполняется на клиенте при загрузке. Он регистрирует обработчик через `ClientPlayNetworking.registerGlobalReceiver()`, который при получении пакета читает boolean и вызывает `BackroomsOverlay.startFade()`. Важно: вызов обёрнут в `client.execute()`, потому что пакет приходит в сетевом потоке, а графика обрабатывается в рендер-потоке. Вызов графических методов из неправильного потока приводит к аварийному завершению.

## 9. Визуальный эффект затемнения — класс `BackroomsOverlay`

Оверлей реализован через событие `HudRenderCallback.EVENT`, которое вызывается каждый кадр рендеринга. Класс хранит четыре статических переменных: `fadeAlpha` (текущая прозрачность от 0.0 до 1.0), `fading` (идёт ли анимация), `fadingIn` (затемнение или осветление), `fadeTimer` (счётчик тиков).

При затемнении `fadeAlpha` плавно увеличивается от 0 до 1 за 40 тиков. При осветлении — уменьшается от 1 до 0. Затем на экран рисуется чёрный прямоугольник размером во весь экран.

Цвет прямоугольника задаётся в формате ARGB: `alpha << 24`, где alpha — значение от 0 до 255, сдвинутое на 24 бита влево (в старший байт 32-битного числа). RGB компоненты равны нулю (чёрный цвет). Когда alpha равен 255, прямоугольник полностью непрозрачен и экран чёрный. Когда alpha равен 0, прямоугольник полностью прозрачен и не виден.

## 10. Звуковая система — класс `BackroomsAmbientSound`

Звуковая система автоматически включает два зацикленных звука (гул ламп и жуткий эмбиент), когда игрок находится в измерении Backrooms, и выключает их при выходе.

Проверка выполняется каждый клиентский тик через `ClientTickEvents.END_CLIENT_TICK`. Текущее измерение определяется сравнением `player.getWorld().getRegistryKey()` с `ModDimensions.BACKROOMS_LEVEL_KEY`.

Звуки реализованы через внутренний класс `BackroomsLoopSound`, наследующий от `MovingSoundInstance`. У него установлено `repeat = true` (зацикленный) и `attenuationType = NONE` (не затухает с расстоянием, слышен одинаково везде). Метод `tick()` обновляет координаты звука до позиции игрока.

Три звуковых события зарегистрированы в `ModSounds.java`, а их маппинг на файлы описан в `assets/backrooms/sounds.json`. Minecraft ищет OGG-файлы по пути, указанному в JSON.

## 11. JSON-ресурсы и текстуры

### 11.1. Blockstates, модели и текстуры

Каждый блок в Minecraft проходит через три уровня JSON для отрисовки. Файл blockstate (`blockstates/backrooms_wall.json`) указывает, какую модель использовать для данного состояния блока. Файл модели (`models/block/backrooms_wall.json`) описывает форму (обычно `minecraft:block/cube_all` — стандартный куб) и ссылку на текстуру. Файл текстуры (`textures/block/backrooms_wall.png`) — это PNG-изображение размером 16×16 пикселей.

### 11.2. Локализация

Файлы `lang/en_us.json` и `lang/ru_ru.json` содержат переводы имён блоков и сущностей. Ключи имеют формат `"block.backrooms.backrooms_wall": "Backrooms Wall"`.

## 12. Тестирование

### 12.1. Используемый фреймворк

Для тестирования используется JUnit 5 (JUnit Jupiter) — стандартный фреймворк юнит-тестирования для Java. Зависимости `junit-jupiter-api` (для написания тестов) и `junit-jupiter-engine` (для их запуска) подключены в `build.gradle` как `testImplementation` и `testRuntimeOnly`.

### 12.2. Структура тестов

Тестовый класс `BackroomsModTest.java` расположен в директории `src/test/java/com/backrooms/mod/` и содержит 10 тестовых методов, каждый аннотированный `@Test` и `@DisplayName` для понятного описания.

Тесты не загружают Minecraft — они проверяют логику, вынесенную из классов мода (хеш-функция, проверки координат, константы). Это позволяет запускать тесты за секунды, без запуска игры.

### 12.3. Описание тестов

Тест 1 — детерминизм хеш-функции. Проверяет, что `mixHash()` при одинаковых аргументах возвращает одинаковый результат. Это критически важно для генерации мира: если хеш недетерминирован, лабиринт будет меняться при каждой загрузке чанка.

Тест 2 — уникальность хешей. Проверяет, что разные координаты дают разные хеши. Если бы близкие координаты давали одинаковые хеши, лабиринт содержал бы одинаковые участки.

Тест 3 — стены на границах сетки. Проверяет, что координаты, кратные 32, являются стенами. Это основной инвариант генератора: границы ячеек сетки всегда заняты блоками стен.

Тест 4 — воздух внутри комнат. Проверяет, что точки далеко от границ сетки (например, (16,16) — центр ячейки) не являются стенами. Если бы этот тест провалился, лабиринт был бы полностью заполнен стенами.

Тест 5 — отрицательные координаты. Проверяет корректность работы с отрицательными координатами. Функция `Math.floorMod()` корректно обрабатывает отрицательные числа (в отличие от оператора `%`, который в Java может возвращать отрицательные значения).

Тест 6 — позиции колонн. Проверяет, что колонны не появляются на произвольных координатах, а только на позициях сетки 10×10 со смещением 5.

Тест 7 — константы телепортации. Проверяет, что задержка телепортации (400 тиков равны 20 секунд) и длительность анимации (40 тиков равны 2 секунды) имеют физический смысл — анимация короче задержки, оба значения положительны.

Тест 8 — валидность MOD_ID. Проверяет, что идентификатор мода содержит только строчные буквы и подчёркивания (требование Minecraft), не пуст и не превышает 64 символа.

Тест 9 — высоты мира. Проверяет логическую корректность вертикальной структуры: пол (Y=0) ниже стен (Y=1-5), стены ниже потолка (Y=6), высота комнаты составляет 5 блоков.

Тест 10 — проходимость лабиринта. Проверяет, что в каждой комнате 32×32 есть хотя бы одна точка, не являющаяся стеной. Это гарантирует, что лабиринт проходим и игрок не окажется замурован.

### 12.4. Запуск тестов

Тесты запускаются командой `.\gradlew.bat test --no-daemon`. Результаты отображаются в консоли, а HTML-отчёт генерируется в `build/reports/tests/test/index.html`.

## 13. Запуск проекта

### 13.1. Требования

Для запуска необходимы: Java Development Kit версии 17 или новее (рекомендуется JDK 21), установленный Gradle (поставляется вместе с проектом через Gradle Wrapper), и доступ к интернету для скачивания зависимостей при первой сборке.

### 13.2. Запуск для разработки

Для запуска Minecraft с модом в режиме разработки используется команда `.\gradlew.bat runClient --no-daemon`. Флаг `--no-daemon` означает, что Gradle не будет запускать фоновый процесс-демон, что экономит оперативную память. При первом запуске Gradle скачает около 500 МБ зависимостей (Minecraft, Fabric, Yarn Mappings). Последующие запуски используют кеш.

### 13.3. Сборка финального JAR

Команда `.\gradlew.bat build --no-daemon` компилирует мод и создаёт файл `backrooms-1.0.0.jar` в директории `build/libs/`. Этот файл можно установить в обычный Minecraft с Fabric Loader, скопировав его в папку `%APPDATA%\.minecraft\mods\`.

## 14. Итоги

Проект демонстрирует применение объектно-ориентированного программирования на Java в контексте модификации видеоигры. Использованы паттерны проектирования: конечный автомат для управления телепортацией, клиент-серверная архитектура с сетевыми пакетами, процедурная генерация мира через детерминистическое хеширование, система наследования для создания сущностей с искусственным интеллектом, событийная модель для тиковых обработчиков и HUD-оверлеев, и cериализация через Codec для интеграции с системой загрузки ресурсов Minecraft.

Тестирование реализовано через JUnit 5 с десятью тестовыми методами, покрывающими критические компоненты системы: генерацию лабиринта, валидацию конфигурации и математическую корректность алгоритмов.
